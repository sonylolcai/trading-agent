# AI K线分析程序设计方案

## 一、程序定位

本程序是一个**辅助决策工具**，而非自动交易系统。程序的核心职责：

1. 获取K线数据并实时绘制K线图
2. 用户点击"提交分析"后，将K线数据发送给AI进行诊断与决策
3. AI返回结构化决策结果，展示给用户
4. 将完整分析记录（含K线数据、AI思考路径、决策结果）保存到本地
5. 由用户手动将记录归类到经验库的正题本或错题本

**程序不执行任何交易。**

三、常驻文件（每次分析均加载，4个）
================================================================

1. 提示词大纲_人设与思维方式.txt
   - 用途：定义AI的身份、思维方式、分析哲学和两阶段工作流程
   - 加载时机：阶段一开头作为System Prompt核心
   - 优先级：最高，所有后续推理的根基

2. 市场诊断框架.txt
   - 用途：三层递进的市场诊断框架
     第一层 周期频谱定位 -> 判断8种市场状态
     第二层 市场背景评估 -> 判断更高时间框架的方向偏好
     第三层 逐K信号确认 -> K线信号验证+寻找入场时机
   - 加载时机：阶段一（诊断阶段）
   - 输出：JSON格式诊断结果

3. 文件16-K线信号识别.txt
   - 用途：K线信号「字典」——信号棒/入场棒/确认棒等微观信号的定义
   - 加载时机：阶段一的逐K信号确认层

4. 文件17-止损和止盈与仓位管理.txt
   - 用途：风控底线——止损设置、仓位计算、交易者方程检验
   - 加载时机：阶段二必附，所有交易决策必须通过其检验

四、条件加载文件（根据诊断结果动态加载，13个）
================================================================

【上涨通道态】诊断结果为 上涨通道(micro_channel/tight_channel/normal_channel/broad_channel) + bullish 时加载：
  5. 上涨通道分析识别.txt    -> 识别HH+HL结构，判断通道类型与演变阶段
  6. 上涨通道交易策略.txt    -> 顺势/逆势交易的具体入场、止损、止盈方案
  7. 文件13-窄通道与宽通道策略.txt -> 通道宽度判断与对应策略调整

【下跌通道态】诊断结果为 下跌通道(micro_channel/tight_channel/normal_channel/broad_channel) + bearish 时加载：
  8. 下跌通道分析识别.txt    -> 识别LL+LH结构，判断通道类型与演变阶段
  9. 下跌通道交易策略.txt    -> 顺势/逆势交易的具体入场、止损、止盈方案
  （通道宽度判断复用文件13-窄通道与宽通道策略.txt）

【极速上涨态】诊断结果为 尖峰/极速上涨(spike) + bullish 时加载：
 10. 极速上涨分析识别.txt    -> 识别尖峰特征，判断高潮与后续演变
 11. 极速上涨交易策略.txt    -> 顺势入场与反转尝试的具体方案

【极速下跌态】诊断结果为 尖峰/极速下跌(spike) + bearish 时加载：
 12. 极速下跌分析识别.txt    -> 识别尖峰特征，判断卖出高潮与后续演变
 13. 极速下跌交易策略.txt    -> 顺势入场与反转尝试的具体方案

【震荡区间态】诊断结果为 交易区间(trading_range) / 趋势型交易区间(trending_tr) 时加载：
 14. 震荡区间分析识别.txt    -> 识别区间边界、突破前兆
 15. 震荡区间交易策略.txt    -> 低买高卖刮头皮的具体方案

【特殊形态】以下文件在检测到对应形态时加载：
 16. 文件14-楔形形态分析交易.txt -> 检测到楔形/三推动时加载
 17. 文件15-二次入场机会.txt     -> 检测到反转尝试时加载

七、JSON异常处理（唯一的风控层）
================================================================

本程序不设硬编码校验规则，风控审核仅包含JSON异常处理。

核心原则：**任何JSON异常，立即报警，输出详细错误调试信息，不进行下一步。**

JSON异常分类与处理：

  a) 语法错误（缺引号、括号不匹配、多余逗号等）
     → 报警，输出原始AI返回文本，不继续

  b) 字段缺失（必需字段如 cycle_position、direction 缺失）
     → 报警，输出缺失字段名 + 已有字段列表，不继续

  c) 值超出范围（confidence不在枚举中、价格为负数等）
     → 报警，输出字段名 + 实际值 + 合法范围，不继续

  d) 纯文本输出（模型拒绝输出JSON，返回自然语言解释）
     → 报警，输出原始文本，不继续

  e) 连续JSON异常（阶段一或阶段二连续出现上述任何异常）
     → 报警，输出连续异常次数 + 历史错误摘要，不继续

报警输出格式：
  ┌──────────────────────────────────────┐
  │  ⚠️ JSON异常报警                      │
  │  异常类型：[a/b/c/d/e]                │
  │  阶段：[阶段一-诊断 / 阶段二-决策]     │
  │  时间：[精确到秒的时间戳]               │
  │  ─────────────────────────────────── │
  │  详细信息：                            │
  │  - 原始AI返回内容：[完整原文]           │
  │  - 解析错误位置：[行号/字符位置]         │
  │  - 缺失字段：[字段列表]（如适用）        │
  │  - 非法值：[字段=值]（如适用）           │
  │  - 合法值范围：[枚举列表]（如适用）       │
  │  - 连续异常次数：[N]（如适用）           │
  │  ─────────────────────────────────── │
  │  处理：流程已终止，请检查后重试          │
  └──────────────────────────────────────┘

八、K线数据格式（带序号）
================================================================

上传给AI的K线数据必须带序号，序号规则：
  - 最新K线序号 = 1
  - 比最新K线早1根 = 2
  - 比最新K线早2根 = 3
  - 以此类推，越早的K线序号越大

示例格式：
  序号 | 时间                | 开盘价  | 最高价  | 最低价  | 收盘价  | 成交量
  -----+--------------------+--------+--------+--------+--------+--------
  1    | 2026-05-18 14:00   | 2650.5 | 2655.2 | 2648.0 | 2653.8 | 12345
  2    | 2026-05-18 13:00   | 2645.0 | 2652.0 | 2643.5 | 2650.5 | 10987
  3    | 2026-05-18 12:00   | 2640.0 | 2646.5 | 2638.0 | 2645.0 | 9876
  ...  | ...                | ...    | ...    | ...    | ...    | ...

序号的作用：
  - 方便AI在分析中精确引用特定K线（如"序号2的K线形成看涨吞没"）
  - 方便用户对照K线图理解AI的分析逻辑
  - 序号从1开始保证"最近的K线最醒目"，符合阅读直觉

九、决策结果格式（阶段二AI输出）
================================================================

AI必须返回以下JSON结构：

  {
    "decision": {
      "order_direction": "做多|做空",
      "order_type": "限价单|突破单|市价单|不下单",
      "entry_price": 0,
      "take_profit_price": 0,
      "stop_loss_price": 0,
      "reasoning": "判断理由的完整叙述",
      "confidence": "high|medium|low",
      "key_factors": [
        "关键因素1",
        "关键因素2"
      ],
      "watch_points": [
        "后续需关注的要点1",
        "后续需关注的要点2"
      ],
      "risk_assessment": "风险评估描述",
      "invalidation_condition": "判断失效条件——什么情况下本决策不再成立"
    },
    "diagnosis_summary": {
      "cycle_position": "阶段一诊断的周期位置",
      "direction": "方向判断",
      "key_signals": ["关键信号列表"]
    }
  }

字段说明：
  - order_direction: 做多/做空/不下单时不填方向
  - order_type:
      限价单 → 在指定价格挂单等待
      突破单 → 价格突破某水平后入场
      市价单 → 当前价格立即入场
      不下单 → 当前不宜入场
  - entry_price: 入场价格（不下单时为0或null）
  - take_profit_price: 止盈目标价（不下单时为0或null）
  - stop_loss_price: 止损价（不下单时为0或null）
  - reasoning: AI的完整判断理由，必须可追溯、可验证
  - confidence: AI对该决策的信心水平
  - key_factors: 影响决策的核心因素（3-5个）
  - watch_points: 入场后需要持续监控的要点
  - risk_assessment: 对该笔交易风险的定性评估
  - invalidation_condition: 判断失效条件——何时该放弃这笔交易

当 order_type = "不下单" 时，reasoning 必须解释为什么当前不宜入场。

10.1 分析记录保存
-----------------
每次"提交分析"完成后，程序自动将完整分析记录保存到本地：

保存路径：records/pending/{YYYY-MM-DD_HH-mm-ss}_{品种}_{周期}.json

记录内容：
  {
    "meta": {
      "timestamp": "分析时间",
      "symbol": "品种代码",
      "timeframe": "K线周期",
      "kline_count": 50,
      "ai_model": "使用的AI模型"
    },
    "kline_data": [
      // 完整的K线数据（带序号），与提交给AI的数据完全一致
    ],
    "stage1_diagnosis": {
      // 阶段一的完整输出（AI原始JSON）
    },
    "stage2_decision": {
      // 阶段二的完整输出（AI原始JSON）
    },
    "ai_thinking_trace": {
      // AI的完整思考路径（如果模型支持输出推理过程）
    },
    "strategy_files_used": [
      // 本轮加载的策略文件列表
    ],
    "experience_loaded": [
      // 本轮加载的经验库条目（如有）
    ]
  }

重要：此时程序不知道这笔交易最终是否成功，记录处于"待归类"状态。

10.2 经验库结构
-----------------
经验库由用户手动维护，程序负责读取：

存储路径：experience/{cycle_position}/
  - success_cases/  → 正题本（用户从pending移入的成功案例）
  - failure_cases/  → 错题本（用户从pending移入的失败案例）

每类最多保留最近20条，阶段二加载最近5条。
若经验库为空或文件不存在，阶段二仍可正常执行（经验库为可选增强，非必须）。

10.3 归类流程
-----------------
  程序保存记录 → records/pending/xxx.json
                      │
                      │ 用户观察市场走势，判断该笔交易是否成功
                      │
                      ├── 成功 → 手动移动到 experience/{cycle_position}/success_cases/xxx.json
                      │
                      └── 失败 → 手动移动到 experience/{cycle_position}/failure_cases/xxx.json

用户可随时清理 pending 目录中不再需要的记录。

  def route_strategy_files(diagnosis):
      cp = diagnosis.get("cycle_position", "unknown")

          files.append("文件13-窄通道与宽通道策略.txt")

      # --- 尖峰态 ---
      elif cp == "spike":
          if dir == "bullish":
              files = ["极速上涨分析识别.txt", "极速上涨交易策略.txt"]
          elif dir == "bearish":
              files = ["极速下跌分析识别.txt", "极速下跌交易策略.txt"]
          else:
              log_warning(f"Spike with neutral direction, no strategy files loaded")

      # --- 震荡区间态 ---
      elif cp in ("trading_range", "trending_tr"):
          files = ["震荡区间分析识别.txt", "震荡区间交易策略.txt"]

      # --- 极端区间／未知 —— 不加载策略文件 ---
      elif cp in ("extreme_tr", "unknown"):
          pass  # 不交易，不加载策略文件

      # --- 特殊形态叠加（基于 detected_patterns 字段） ---
      if "wedge" in patterns:
          files.append("文件14-楔形形态分析交易.txt")
      if "reversal_attempt" in patterns:
          files.append("文件15-二次入场机会.txt")

      return files

十二、程序实现要点
================================================================

1. K线数据获取与展示
   - 程序启动后持续获取K线数据
   - 实时绘制K线图（建议使用 mplfinance / lightweight-charts 等）
   - K线图展示最近30-50根K线
   - 在K线图旁展示序号标注（序号1=最新，方便用户对照AI分析）

2. Token管理
   - 阶段一约 30K tokens input（3个常驻文件+K线数据）
   - 阶段二约 20-35K tokens input（1-3个策略文件+风控+经验）
   - 建议使用API caching机制缓存不变的System Prompt

3. 文件读取顺序
   - 提示词文件必须按指定顺序拼接为System Prompt
   - 顺序设计逻辑：
     人设(身份) -> 诊断框架(看什么) -> K线信号(怎么看) -> 策略文件(怎么做) -> 风控(底线在哪)
   - 不可随意调换顺序，否则AI认知链路断裂
   - 每个文件都假设前面的文件已加载

4. Token预算管理
   - 模型最低要求：32K上下文窗口
   - 阶段一输入预算：约30K token
   - 阶段二输入预算：约20-35K token
   - 单次输出上限：8K token
   - Token超限处理：
     * 若输入超限：优先保留最近20根K线完整数据，截断更早的历史数据
     * 若输出超限：阶段一只输出JSON诊断结果，不输出分析报告文本
   - 会话级监控：单会话累计消耗超过200K token时提示用户

5. 经验库加载逻辑
   - "最近"的排序依据为交易发生日期（非文件写入时间），降序排列
   - 若不足5条则全部加载，若为空则跳过
   - 经验库文件不存在时，阶段二仍可正常执行
